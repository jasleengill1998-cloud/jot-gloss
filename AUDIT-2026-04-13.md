# Jot Gloss Technical Audit

**Date:** 2026-04-13  
**Scope:** Front-end React/Vite app, local persistence model, and `/api/sync` cloud sync endpoint.  
**Method:** Static code review + production build verification.

## Executive Summary

The app builds successfully and core architecture is coherent, but there are **three high-priority security/data-risk issues** and **two medium-priority reliability concerns**:

1. **Unauthenticated cloud sync endpoint** allows anyone who can hit `/api/sync` to read/write the full synced dataset.
2. **Permissive CORS (`*`)** on that same endpoint increases risk of cross-origin abuse.
3. **Runtime code execution via `new Function`** in the file preview path can execute untrusted content.
4. **Non-atomic cloud writes** can lose updates in multi-device scenarios (last-write-wins with no version guard).
5. **Potential memory/resource growth** from blob listing + full payload uploads as dataset scales.

---

## Findings

### 1) Unauthenticated cloud sync endpoint (High)

`/api/sync` accepts `GET` and `POST` and performs reads/writes using a server-side Blob RW token, but it does not authenticate the caller (no session check, API key, or signature requirement).【F:api/sync.ts†L21-L113】  
This means any party able to access the deployed endpoint can attempt to pull or overwrite synced data.

**Recommendation:**
- Require caller authentication before allowing `GET`/`POST`.
- At minimum, gate the endpoint behind an app-level shared secret header and rotate regularly.
- Prefer per-user scoped auth and namespaced blob paths.

### 2) CORS wildcard on sync API (High)

The endpoint returns `access-control-allow-origin: *` for both normal responses and preflight handling.【F:api/sync.ts†L14-L17】【F:api/sync.ts†L23-L30】  
Combined with missing auth, this broadens abuse surface for browser-based cross-origin calls.

**Recommendation:**
- Restrict `access-control-allow-origin` to known production origins.
- Consider dropping CORS entirely if same-origin requests are sufficient.

### 3) Dynamic code execution in viewer sandbox (High)

`FileViewer` evaluates code with `new Function(...)` during component extraction/render preparation.【F:src/components/FileViewer.tsx†L248】  
Even with iframe sandboxing patterns, this is still a high-risk primitive if content can come from untrusted sources (clipboard/import/cloud pull).

**Recommendation:**
- Treat all imported content as untrusted.
- Disable executable previews by default, or require explicit trust confirmation.
- If execution is required, use a stronger isolation boundary (e.g., dedicated origin + strict sandbox/CSP, no parent privileges).

### 4) Last-write-wins sync without conflict control (Medium)

The sync flow overwrites a single blob object (`studybloom-sync/all-files.json`) and merges by timestamps; there is no optimistic concurrency token/etag guard in writes.【F:api/sync.ts†L11-L12】【F:api/sync.ts†L74-L95】【F:src/utils/cloudSync.ts†L61-L86】  
Concurrent device updates can silently clobber each other.

**Recommendation:**
- Add version/etag preconditions for writes.
- Track per-record version vectors or server-issued revision IDs.
- Surface conflict resolution UI for same-record divergent edits.

### 5) Scale/performance risk in sync data model (Medium)

The API lists blobs by prefix and stores all files in one JSON payload; client pushes the full dataset for updates.【F:api/sync.ts†L46-L56】【F:api/sync.ts†L74-L95】【F:src/utils/cloudSync.ts†L34-L47】  
As data grows, this can increase latency, memory usage, and failure probability.

**Recommendation:**
- Move to chunked/object-per-file sync or batched deltas.
- Avoid prefix-list on every read when object path is known.
- Add payload size limits and telemetry.

## Priority Remediation Plan

1. Add endpoint authentication + origin restrictions first.
2. Reduce/remove runtime code execution surface in viewer path.
3. Introduce revision-safe sync writes.
4. Redesign sync storage model for incremental updates.
